ทำความเข้าใจ Container, Image และความแตกต่างจาก Virtual Machine

🐳 Docker คืออะไร?

Docker คือแพลตฟอร์มสำหรับพัฒนา, ส่ง, และรันแอปพลิเคชันใน Container ซึ่งเป็นสภาพแวดล้อมที่แยกอิสระ (isolated environment)

  • Portable — รันได้ทุกที่ที่มี Docker ติดตั้ง
  • Consistent — "works on my machine" หมดปัญหา!
  • Lightweight — เบากว่า VM มาก ใช้ทรัพยากรน้อยกว่า
  • Fast — เริ่มต้นใน milliseconds ไม่ใช่ minutes
  • Scalable — ขยายได้ง่ายด้วย Docker Compose/K8s

🧱 แนวคิดหลัก 3 อย่าง

🖼️ Image
แม่แบบ (blueprint) ของ Container — read-only, เปรียบได้กับ "คลาส" ใน OOP
📦 Container
Instance ที่รันได้จาก Image — เปรียบได้กับ "object" ที่สร้างจากคลาส
📋 Dockerfile
สคริปต์คำสั่งสำหรับสร้าง Image — ทีละขั้นตอน

Docker Image — แม่แบบของ Container

ทำความเข้าใจ Image, Layers และคำสั่งจัดการ Image

🐳
Image ประกอบด้วย Layers ที่ซ้อนกัน แต่ละ Instruction ใน Dockerfile สร้าง Layer ใหม่ — Layer ที่เหมือนกันจะถูก Cache และแชร์กันระหว่าง Image ต่างๆ ทำให้ประหยัด disk space

📦 Image Layers

Layer 4: COPY app files ← เปลี่ยนบ่อย
Layer 3: RUN npm install
Layer 2: COPY package.json
Layer 1: FROM node:20-alpine ← Base image, เปลี่ยนน้อย

แต่ละ Layer จะถูก Cache ไว้ ถ้า Layer ล่างไม่เปลี่ยน Layer บนจะใช้ Cache — ทำให้ build เร็วขึ้นมาก

🛠️ คำสั่งจัดการ Image

# ดาวน์โหลด Image จาก Docker Hub
docker pull nginx
docker pull node:20-alpine
docker pull postgres:16

# ดู Image ทั้งหมดที่มี
docker images
docker image ls

# ดูรายละเอียด Image
docker inspect nginx
docker image inspect nginx

# ดู Layer ของ Image
docker history nginx

# ลบ Image
docker rmi nginx
docker image rm nginx

# ลบ Image ที่ไม่ได้ใช้ทั้งหมด (dangling)
docker image prune

# ลบ Image ที่ไม่ได้ใช้ทั้งหมด (รวม unreferenced)
docker image prune -a

# Tag Image
docker tag myapp:latest myapp:v1.0.0
docker tag myapp:latest registry.example.com/myapp:latest

🔍 Image Naming Convention

# รูปแบบชื่อ Image
[registry/][username/]image-name[:tag]

# ตัวอย่าง
nginx                          # Official image, tag = latest
nginx:1.25                     # Official image, tag = 1.25
node:20-alpine                 # Official image, alpine variant
username/myapp:v1.0            # User image บน Docker Hub
ghcr.io/org/app:sha-abc123     # GitHub Container Registry
localhost:5000/myapp:dev       # Local registry
💡
แนะนำใช้ tag เฉพาะเจาะจง เสมอ เช่น node:20-alpine แทน node:latest เพื่อให้ build reproducible และปลอดภัยกว่า

Docker Container — รัน Application

คำสั่ง docker run และการจัดการ Container

🚀 docker run — คำสั่งสำคัญที่สุด

# รูปแบบพื้นฐาน
docker run [OPTIONS] IMAGE [COMMAND]

# ตัวอย่างพื้นฐาน
docker run nginx

# -d : รันเบื้องหลัง (detached mode)
docker run -d nginx

# -p : map port [host]:[container]
docker run -d -p 8080:80 nginx

# --name : ตั้งชื่อ Container
docker run -d -p 8080:80 --name my-nginx nginx

# -e : ส่ง Environment Variable
docker run -d -e MYSQL_ROOT_PASSWORD=secret mysql:8

# -v : Mount Volume [host-path]:[container-path]
docker run -d -v ./data:/var/lib/mysql mysql:8

# --rm : ลบ Container อัตโนมัติเมื่อหยุด
docker run --rm ubuntu echo "Hello"

# -it : Interactive + TTY (เข้า shell ได้)
docker run -it ubuntu bash

# --network : เชื่อม Network
docker run --network my-network myapp

# รวมหลาย flag
docker run -d --name webserver -p 8080:80 -v ./html:/usr/share/nginx/html nginx

🛠️ คำสั่งจัดการ Container

# ดู Container ที่รันอยู่
docker ps
docker ps --format "table {{.Names}}\t{{.Status}}\t{{.Ports}}"

# ดู Container ทั้งหมด (รวมที่หยุดแล้ว)
docker ps -a

# หยุด Container
docker stop my-nginx

# เริ่ม Container ที่หยุด
docker start my-nginx

# Restart Container
docker restart my-nginx

# ลบ Container (ต้องหยุดก่อน)
docker rm my-nginx

# หยุดและลบพร้อมกัน
docker rm -f my-nginx

# ดู Log ของ Container
docker logs my-nginx
docker logs -f my-nginx        # Follow (realtime)
docker logs --tail 50 my-nginx  # แสดง 50 บรรทัดล่าสุด

# เข้าไปใน Container ที่รันอยู่
docker exec -it my-nginx bash
docker exec -it my-nginx sh    # สำหรับ Alpine (ไม่มี bash)

# รันคำสั่งใน Container
docker exec my-nginx nginx -t   # ทดสอบ config nginx

# ดูสถิติการใช้ทรัพยากร
docker stats
docker stats my-nginx

# Copy ไฟล์ระหว่าง host กับ container
docker cp my-nginx:/etc/nginx/nginx.conf ./nginx.conf
docker cp ./nginx.conf my-nginx:/etc/nginx/nginx.conf

# ลบ Container ที่หยุดทั้งหมด
docker container prune

docker create → Running
docker start/run → Paused
docker pause → Running

Dockerfile — สร้าง Image ของตัวเอง

Instruction ทั้งหมดใน Dockerfile และตัวอย่างจริง

📋 Dockerfile Instructions ทั้งหมด

Instructionความหมายตัวอย่าง
FROMBase Image (ต้องมีเป็น Instruction แรก)FROM node:20-alpine
WORKDIRตั้ง Working Directory ภายใน ContainerWORKDIR /app
COPYCopy ไฟล์จาก host → containerCOPY . .
ADDเหมือน COPY แต่รองรับ URL และ .tarADD app.tar.gz /app
RUNรันคำสั่งตอน build (สร้าง layer)RUN npm install
CMDคำสั่ง default เมื่อ Container เริ่ม (override ได้)CMD ["node","server.js"]
ENTRYPOINTคำสั่งหลักของ Container (override ยากกว่า)ENTRYPOINT ["npm"]
ENVตั้ง Environment VariableENV NODE_ENV=production
ARGตัวแปรตอน build เท่านั้นARG APP_VERSION=1.0
EXPOSEประกาศ port (เป็นแค่ documentation)EXPOSE 3000
VOLUMEสร้าง mount point สำหรับ volumeVOLUME ["/data"]
USERตั้ง user ที่รันคำสั่งUSER node
HEALTHCHECKตรวจสอบสุขภาพ ContainerHEALTHCHECK CMD curl -f http://localhost/
LABELเพิ่ม metadata ให้ ImageLABEL version="1.0"

🟢 Dockerfile สำหรับ Node.js App

# ใช้ Alpine เพื่อให้ Image เล็ก
FROM node:20-alpine

# ตั้ง Working Directory
WORKDIR /app

# Copy package files ก่อน (ใช้ประโยชน์จาก Layer Cache)
COPY package.json package-lock.json ./

# ติดตั้ง dependencies
RUN npm ci --only=production

# Copy source code
COPY . .

# ประกาศ port
EXPOSE 3000

# Health Check
HEALTHCHECK --interval=30s --timeout=3s \
  CMD wget -qO- http://localhost:3000/health || exit 1

# ใช้ non-root user (Security Best Practice)
USER node

# คำสั่งเริ่มต้น
CMD ["node", "server.js"]
# Build Image จาก Dockerfile
docker build -t myapp:latest .

# Build พร้อมระบุ Dockerfile
docker build -f Dockerfile.prod -t myapp:prod .

# Build พร้อม ARG
docker build --build-arg APP_VERSION=2.0 -t myapp:v2.0 .

# Build โดยไม่ใช้ Cache
docker build --no-cache -t myapp:latest .

🐍 Dockerfile สำหรับ Python (FastAPI)

FROM python:3.12-slim

WORKDIR /app

# ติดตั้ง system dependencies
RUN apt-get update && apt-get install -y \
    gcc \
    && rm -rf /var/lib/apt/lists/*

# Copy requirements ก่อน (layer cache)
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

EXPOSE 8000

CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]

Volume — จัดการข้อมูลถาวร

เก็บข้อมูลให้อยู่รอดแม้ Container จะถูกลบ

⚠️
ข้อมูลใน Container จะหายไปทันทีเมื่อ Container ถูกลบ ถ้าต้องการเก็บข้อมูลถาวร (เช่น database) ต้องใช้ Volume

📦 ประเภทของ Volume/Mount

🗂️ Named Volume
Docker จัดการตำแหน่งเอง
แชร์ระหว่าง Container ได้
→ แนะนำสำหรับ Database
📁 Bind Mount
Map folder บน host
เห็นการเปลี่ยนแปลง real-time
→ แนะนำสำหรับ Development
💾 tmpfs Mount
เก็บใน RAM
เร็วมาก แต่หายเมื่อหยุด
→ สำหรับข้อมูลชั่วคราว

🛠️ คำสั่งจัดการ Volume

# สร้าง Named Volume
docker volume create my-data

# ดู Volume ทั้งหมด
docker volume ls

# ดูรายละเอียด Volume
docker volume inspect my-data

# ลบ Volume
docker volume rm my-data

# ลบ Volume ที่ไม่ได้ใช้
docker volume prune

💡 การใช้งาน Volume จริง

# Named Volume — สำหรับ MySQL
docker run -d \
  --name mysql-db \
  -e MYSQL_ROOT_PASSWORD=secret \
  -e MYSQL_DATABASE=myapp \
  -v mysql-data:/var/lib/mysql \   # Named Volume
  -p 3306:3306 \
  mysql:8

# Bind Mount — สำหรับ Development (Hot Reload)
docker run -d \
  --name myapp-dev \
  -v $(pwd):/app \                # Bind Mount
  -v /app/node_modules \          # Anonymous Volume (ไม่ทับ node_modules)
  -p 3000:3000 \
  myapp:dev

# Read-only Bind Mount (ปลอดภัยกว่า)
docker run -v $(pwd)/config:/app/config:ro myapp

# tmpfs Mount
docker run --tmpfs /tmp myapp

Docker Network — เชื่อมต่อ Container

ทำให้ Container คุยกันได้และควบคุม Network

🌐 ประเภทของ Network Driver

Driverลักษณะใช้เมื่อ
bridge (default)Virtual network บน host, Container คุยกันได้ผ่าน container nameContainer หลายตัวบน host เดียว
hostใช้ network ของ host โดยตรง ไม่ isolatePerformance สูงสุด, Linux เท่านั้น
noneไม่มี network เลยContainer ที่ไม่ต้องการ network
overlayเชื่อม Docker Swarm หลาย hostMulti-host container orchestration
macvlanContainer มี MAC address ของตัวเองLegacy apps ที่ต้องการ IP จริง

🛠️ คำสั่ง Network

# ดู Network ทั้งหมด
docker network ls

# สร้าง Network
docker network create my-network
docker network create --driver bridge my-network

# ดูรายละเอียด Network
docker network inspect my-network

# เพิ่ม Container เข้า Network
docker network connect my-network my-container

# ถอด Container ออกจาก Network
docker network disconnect my-network my-container

# ลบ Network
docker network rm my-network
docker network prune  # ลบที่ไม่ได้ใช้

🔗 Container คุยกันผ่าน Network

# สร้าง network ก่อน
docker network create app-network

# รัน database ใน network
docker run -d \
  --name postgres-db \
  --network app-network \
  -e POSTGRES_PASSWORD=secret \
  postgres:16

# รัน app ใน network เดียวกัน
# สามารถใช้ "postgres-db" เป็น hostname ได้เลย!
docker run -d \
  --name myapp \
  --network app-network \
  -e DATABASE_URL=postgresql://postgres:secret@postgres-db:5432/mydb \
  -p 3000:3000 \
  myapp:latest

# Container ใน network เดียวกัน ใช้ชื่อ container เป็น hostname ได้
# postgres-db → resolves to container's IP อัตโนมัติ
💡
ใน Custom Bridge Network (ไม่ใช่ default bridge) Docker จะ resolve ชื่อ Container เป็น IP อัตโนมัติ (DNS-based service discovery) — ไม่ต้องรู้ IP จริงของ Container

Docker Compose — จัดการหลาย Container

นิยาม Application Stack ด้วยไฟล์ YAML ไฟล์เดียว

🐳
Docker Compose ช่วยให้นิยาม Service ทั้งหมด (web, database, cache, queue) ไว้ในไฟล์ docker-compose.yml เดียว และรัน/หยุดทั้งหมดด้วยคำสั่งเดียว

📄 docker-compose.yml พื้นฐาน

# docker-compose.yml
version: '3.9'

services:
  # Service ที่ 1: Web App (Node.js)
  web:
    build: .                      # Build จาก Dockerfile ในโฟลเดอร์ปัจจุบัน
    ports:
      - "3000:3000"
    environment:
      - NODE_ENV=development
      - DATABASE_URL=postgresql://user:pass@db:5432/mydb
    volumes:
      - .:/app                    # Bind Mount สำหรับ Dev
      - /app/node_modules
    depends_on:
      db:
        condition: service_healthy  # รอ db healthy ก่อน
    networks:
      - app-network
    restart: unless-stopped

  # Service ที่ 2: PostgreSQL
  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: mydb
    volumes:
      - postgres-data:/var/lib/postgresql/data
      - ./init.sql:/docker-entrypoint-initdb.d/init.sql  # รัน SQL ตอนเริ่ม
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U user -d mydb"]
      interval: 10s
      timeout: 5s
      retries: 5
    networks:
      - app-network

  # Service ที่ 3: Redis Cache
  redis:
    image: redis:7-alpine
    command: redis-server --appendonly yes
    volumes:
      - redis-data:/data
    networks:
      - app-network
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 3s
      retries: 3

volumes:
  postgres-data:   # Named Volume
  redis-data:

networks:
  app-network:
    driver: bridge

🛠️ คำสั่ง Docker Compose

# เริ่ม services ทั้งหมด (พร้อม build)
docker compose up
docker compose up -d                # Background mode
docker compose up --build           # Force rebuild
docker compose up -d --build        # Background + rebuild

# หยุด services
docker compose down
docker compose down -v             # หยุดพร้อมลบ Volume ด้วย!
docker compose down --rmi all     # หยุดพร้อมลบ Image

# รัน service เฉพาะตัว
docker compose up -d db redis

# ดู Logs
docker compose logs
docker compose logs -f             # Follow
docker compose logs -f web        # Log เฉพาะ service

# เข้า Shell ใน Service
docker compose exec web bash
docker compose exec db psql -U user -d mydb

# รัน one-off command
docker compose run --rm web npm test
docker compose run --rm web npx prisma migrate

# ดูสถานะ Services
docker compose ps

# Restart service
docker compose restart web

# Scale service
docker compose up --scale web=3  # รัน 3 instances

🗂️ หลาย Compose File (Dev vs Prod)

# docker-compose.yml — Base config
services:
  web:
    image: myapp:latest
    environment:
      - NODE_ENV=production

# docker-compose.override.yml — Dev overrides (อ่านอัตโนมัติ)
services:
  web:
    build: .
    volumes:
      - .:/app
    environment:
      - NODE_ENV=development

# docker-compose.prod.yml — Production config
services:
  web:
    restart: always
    deploy:
      replicas: 3
# Development (อ่าน base + override อัตโนมัติ)
docker compose up

# Production (ระบุไฟล์เอง)
docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d

Docker Registry — แชร์ Image

Push และ Pull Image ไปยัง Docker Hub และ Registry อื่นๆ

🌐 Docker Hub — Registry สาธารณะ

# Login เข้า Docker Hub
docker login
# กรอก Username และ Password / Access Token

# Tag Image สำหรับ push
docker tag myapp:latest username/myapp:latest
docker tag myapp:latest username/myapp:v1.0.0

# Push ขึ้น Docker Hub
docker push username/myapp:latest
docker push username/myapp:v1.0.0

# Pull จาก Docker Hub
docker pull username/myapp:latest

# Logout
docker logout

🔐 GitHub Container Registry (GHCR)

# Login ด้วย GitHub Token
echo $GITHUB_TOKEN | docker login \
  ghcr.io -u USERNAME \
  --password-stdin

# Tag
docker tag myapp:latest \
  ghcr.io/username/myapp:latest

# Push
docker push ghcr.io/username/myapp:latest

☁️ AWS ECR

# Login ด้วย AWS CLI
aws ecr get-login-password \
  --region ap-southeast-1 | \
  docker login --username AWS \
  --password-stdin \
  ACCOUNT.dkr.ecr.REGION.amazonaws.com

# Tag และ Push
docker tag myapp:latest \
  ACCOUNT.dkr.ecr.REGION.amazonaws.com/myapp:latest

docker push \
  ACCOUNT.dkr.ecr.REGION.amazonaws.com/myapp:latest

🏠 Local Registry (Self-hosted)

# รัน Local Registry บน port 5000
docker run -d \
  --name registry \
  -p 5000:5000 \
  -v registry-data:/var/lib/registry \
  registry:2

# Tag สำหรับ Local Registry
docker tag myapp:latest localhost:5000/myapp:latest

# Push ไป Local Registry
docker push localhost:5000/myapp:latest

# Pull จาก Local Registry
docker pull localhost:5000/myapp:latest

Multi-stage Build — Image เล็กและปลอดภัย

แยก Build Environment ออกจาก Production Environment

💡
Multi-stage Build ทำให้ Production Image ไม่มี build tools, source code, dependencies ที่ไม่จำเป็น — ทำให้ Image เล็กลงมาก และปลอดภัยกว่า

🔨 Multi-stage สำหรับ Node.js

❌ Single-stage (ใหญ่)
FROM node:20

WORKDIR /app
COPY . .
RUN npm install
RUN npm run build

CMD ["node", "dist/server.js"]

# ขนาด Image: ~1.2 GB 😱
# มี devDependencies ทั้งหมด
# มี source code
# มี compiler/tools
✅ Multi-stage (เล็ก)
# Stage 1: Build
FROM node:20-alpine AS builder

WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Stage 2: Production
FROM node:20-alpine AS production

WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY --from=builder /app/dist ./dist
USER node
CMD ["node", "dist/server.js"]

# ขนาด Image: ~180 MB ✅

🦀 Multi-stage สำหรับ Go / Compiled Languages

# Stage 1: Build (มี compiler)
FROM golang:1.22-alpine AS builder

WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download

COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -o server ./cmd/server

# Stage 2: Production (แค่ binary ไฟล์เดียว!)
FROM scratch                           # Image ว่างเปล่า 0 bytes!

COPY --from=builder /app/server /server
COPY --from=builder /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/

EXPOSE 8080
ENTRYPOINT ["/server"]

# ขนาด Image: ~15 MB 🎉 (แค่ binary + SSL certs)

⚛️ Multi-stage สำหรับ React (Frontend)

# Stage 1: Build React app
FROM node:20-alpine AS builder

WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
ARG VITE_API_URL
RUN npm run build

# Stage 2: Serve ด้วย Nginx
FROM nginx:alpine

# Copy เฉพาะ static files ที่ build แล้ว
COPY --from=builder /app/dist /usr/share/nginx/html

# Custom nginx config สำหรับ SPA routing
COPY nginx.conf /etc/nginx/conf.d/default.conf

EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]

# ขนาด Image: ~50 MB (nginx + static files เท่านั้น)

Environment Variables — ตั้งค่า Container

จัดการ Configuration อย่างปลอดภัยด้วย Env Vars และ Secrets

🔧 วิธีส่ง Environment Variable

# วิธีที่ 1: -e flag (inline)
docker run -e NODE_ENV=production -e PORT=3000 myapp

# วิธีที่ 2: --env-file (แนะนำ)
docker run --env-file .env myapp

# .env file
NODE_ENV=production
PORT=3000
DATABASE_URL=postgresql://user:pass@localhost:5432/mydb
JWT_SECRET=super-secret-key
REDIS_URL=redis://localhost:6379

📄 Env Vars ใน Docker Compose

# docker-compose.yml
services:
  web:
    image: myapp:latest
    environment:           # วิธีที่ 1: กำหนดตรงๆ
      NODE_ENV: production
      PORT: 3000
      DATABASE_URL: ${DATABASE_URL}  # อ่านจาก .env ของ host

    env_file:             # วิธีที่ 2: อ่านจากไฟล์
      - .env
      - .env.production

  db:
    image: postgres:16
    environment:
      POSTGRES_USER: ${DB_USER:-postgres}      # Default value
      POSTGRES_PASSWORD: ${DB_PASSWORD:?error} # Required! Error ถ้าไม่มี
      POSTGRES_DB: ${DB_NAME:-myapp}
# .env — ไม่ commit ขึ้น Git!
DB_USER=myuser
DB_PASSWORD=supersecret
DB_NAME=myapp
DATABASE_URL=postgresql://myuser:supersecret@db:5432/myapp
🚨
เพิ่ม .env ใน .gitignore เสมอ! อย่า commit ข้อมูล sensitive ขึ้น Git เด็ดขาด สร้าง .env.example ที่ไม่มีค่าจริงแทน

🔒 Docker Secrets (Production)

# สร้าง Secret (Docker Swarm)
echo "my-super-secret" | docker secret create db_password -

# ใช้ใน docker-compose.yml
services:
  db:
    image: postgres:16
    secrets:
      - db_password
    environment:
      POSTGRES_PASSWORD_FILE: /run/secrets/db_password

secrets:
  db_password:
    external: true  # ใช้ secret ที่สร้างไว้แล้ว

Best Practices & เทคนิคที่ควรรู้

วิธีปฏิบัติที่ดีเพื่อ Image เล็ก, ปลอดภัย และ Maintainable

📦 Best Practices: Image

  • ใช้ Official Image เป็น Base เสมอ
  • ใช้ Alpine/slim variant เพื่อลดขนาด
  • ระบุ tag เฉพาะ ไม่ใช้ :latest ใน production
  • ใช้ Multi-stage Build สำหรับ compiled languages
  • รวม RUN commands ด้วย && เพื่อลด Layer
  • ลบ cache หลังติดตั้ง package
  • เพิ่ม .dockerignore ทุกโปรเจค

🔐 Best Practices: Security

  • ไม่ run container เป็น root user
  • ไม่ hardcode secrets/passwords ใน Dockerfile
  • ใช้ USER instruction เพื่อ non-root user
  • Scan Image ด้วย docker scout หรือ Trivy
  • ใช้ Read-only filesystem: --read-only
  • จำกัด capabilities: --cap-drop ALL
  • อัปเดต Base Image สม่ำเสมอ

⚡ เทคนิค Optimize Layer Cache

❌ Cache miss ทุกครั้ง
FROM node:20-alpine
WORKDIR /app
# Copy ทุกอย่างก่อน
COPY . .
# install หลัง — invalidate cache
# ทุกครั้งที่โค้ดเปลี่ยน!
RUN npm install
CMD ["node", "server.js"]
✅ ใช้ Cache ได้ดี
FROM node:20-alpine
WORKDIR /app
# Copy package files ก่อน
COPY package*.json ./
# Install — Cache ถ้า package ไม่เปลี่ยน
RUN npm ci
# Copy code หลังสุด
COPY . .
CMD ["node", "server.js"]

docker network prune    # Network ที่ไม่ได้ใช้
